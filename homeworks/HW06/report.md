## 1. Dataset

- Выбран датасет: `S06-hw-dataset-01.csv` (бинарная классификация).
- Размер: **12000** строк × **30** столбцов.
- Целевая переменная: `target`, принимает значения 0 и 1.
  - Класс 0: **8119** объектов (**67.7%**).
  - Класс 1: **3881** объектов (**32.3%**).
- Признаки:
  - 24 числовых признака: `num01` … `num24` (тип `float64`).
  - 4 целочисленных признака, интерпретируемых как категориальные/дискретные:  
    `cat_contract`, `cat_region`, `cat_payment`, `tenure_months` (тип `int64`).
  - Столбец `id` используется только как идентификатор и исключён из матрицы признаков `X`.
- Пропуски в данных отсутствуют во всех столбцах.

## 2. Protocol

- Разбиение на выборки:
  - Использован `train_test_split`.
  - Размеры: `X_train` – 9000 объектов (75%), `X_test` – 3000 объектов (25%).
  - Параметры: `test_size = 0.25`, `random_state = 42`, `stratify = y` для сохранения долей классов.
- Подбор гиперпараметров:
  - Выполняется только на обучающей выборке (`X_train`, `y_train`).
  - Используется `GridSearchCV` с **5-fold cross-validation** (`cv = 5`).
  - Целевая метрика для подбора: **ROC-AUC**.
- Метрики на отложенной выборке (test):
  - **Accuracy** — доля правильно классифицированных объектов; удобна для общей оценки качества.
  - **F1-score** — гармоническое среднее precision и recall для положительного класса 1; важна из-за несбалансированности классов.
  - **ROC-AUC** — площадь под ROC-кривой; устойчива к выбору порога и отражает способность модели ранжировать объекты класса 1 выше, чем класса 0.
- Все окончательные оценки качества моделей считаются только один раз на **test** после подбора гиперпараметров на **train**.

## 3. Models

В эксперименте сравнивались следующие модели.

**Бейзлайны**

- `DummyClassifier` (baseline):
  - Стратегия: `stratified` (случайные прогнозы с сохранением долей классов).
  - Служит минимальной отправной точкой, ниже которой качество быть не должно.

- `LogisticRegression` (baseline из S05):
  - Пайплайн: `StandardScaler` → `LogisticRegression`.
  - Параметры: `random_state = 42`, `max_iter = 1000`.
  - Линейная модель, чувствительная к нелинейностям в данных.

**Модели с подбором гиперпараметров (GridSearchCV, cv=5, scoring='roc_auc')**

- `DecisionTreeClassifier`:
  - Базовая модель: `DecisionTreeClassifier(random_state = 42)`.
  - Подбираемые гиперпараметры:
    - `max_depth: [3, 5, 7, 10, 15]`.
    - `min_samples_leaf: [1, 2, 5, 10]`.
  - Лучшие параметры по ROC-AUC (CV):
    - `max_depth = 10`.
    - `min_samples_leaf = 10`.

- `RandomForestClassifier`:
  - Базовая модель: `RandomForestClassifier(random_state = 42, n_jobs = -1)`.
  - Подбираемые гиперпараметры:
    - `n_estimators: [50, 100, 150]`.
    - `max_depth: [10, 15, 20]`.
    - `min_samples_leaf: [2, 5]`.
  - Лучшие параметры по ROC-AUC (CV):
    - `n_estimators = 150`.
    - `max_depth = 20`.
    - `min_samples_leaf = 2`.

- `GradientBoostingClassifier` (boosting):
  - Базовая модель: `GradientBoostingClassifier(random_state = 42)`.
  - Подбираемые гиперпараметры:
    - `n_estimators: [50, 100, 150]`.
    - `learning_rate: [0.01, 0.05, 0.1]`.
    - `max_depth: [3, 5, 7]` (глубина базовых деревьев).
  - Лучшие параметры по ROC-AUC (CV):
    - `n_estimators = 150`.
    - `learning_rate = 0.1`.
    - `max_depth = 7`.

- `AdaBoostClassifier` (boosting):
  - Базовая модель: `AdaBoostClassifier(random_state = 42)`.
  - Подбираемые гиперпараметры:
    - `n_estimators: [50, 100, 150]`.
    - `learning_rate: [0.5, 1.0, 1.5]`.
  - Лучшие параметры по ROC-AUC (CV):
    - `n_estimators = 150`.
    - `learning_rate = 1.5`.

Стэкинг в данной реализации не использовался; основное внимание уделено сравнению деревьев и классических ансамблей (bagging и boosting).

## 4. Results

Финальные метрики на отложенной выборке (test):

- RandomForestClassifier показывает сравнимое качество, но немного уступает GradientBoosting по всем трём метрикам.

## 5. Analysis

### Устойчивость к random_state

- В основном эксперименте использован фиксированный `random_state = 42` для разбиения train/test и инициализации моделей.
- Для более детальной оценки устойчивости следовало бы повторить обучение 1–2 ключевых моделей (например, RandomForest и GradientBoosting) при нескольких значениях `random_state` и сравнить разброс ROC-AUC на test.
- Ожидается, что одиночное дерево будет более чувствительно к выбору `random_state`, тогда как ансамбли (RandomForest и GradientBoosting) благодаря усреднению и последовательному бустингу должны демонстрировать меньший разброс качества.

### Ошибки (confusion matrix для лучшей модели)

Для **GradientBoostingClassifier** на тестовой выборке (3000 объектов) получена следующая матрица ошибок:

- True Negatives (TN)  = 1964 — объекты класса 0, предсказанные как 0.
- False Positives (FP) = 66   — объекты класса 0, ошибочно отнесённые к классу 1.
- False Negatives (FN) = 129  — объекты класса 1, ошибочно отнесённые к классу 0.
- True Positives (TP)  = 841  — объекты класса 1, корректно предсказанные как 1.

Интерпретация:

- Модель редко даёт ложные срабатывания (FP относительно невелико), то есть не слишком часто неправильно помечает клиентов класса 0 как класс 1.
- Часть позитивных объектов всё же теряется (FN = 129), однако при этом достигается высокий recall и F1-score, что говорит о разумном балансе между пропусками и ложными срабатываниями.

### Интерпретация (Permutation Importance)

- Существенный вклад также дают `num07`, `num04`, `num24` и несколько других числовых признаков; это основные носители сигналов для модели.

## 6. Conclusion

1. Дерево решений даёт заметный прирост к линейному бейзлайну, но одиночная модель остаётся менее устойчивой и менее точной, чем ансамбли.
2. Bagging в виде RandomForest значительно улучшает качество по сравнению с одним деревом за счёт усреднения по многим деревьям и снижения разброса (variance).
3. Boosting (GradientBoosting, AdaBoost) позволяет ещё сильнее повысить качество, последовательно исправляя ошибки предыдущих деревьев; в данном эксперименте именно GradientBoosting даёт наилучший ROC-AUC и F1-score.
4. Честный ML-протокол с раздельными train/test, подбором гиперпараметров только на train и использованием CV критичен: он позволяет избежать утечки информации и завышения метрик.
5. Permutation importance показывает, какие признаки действительно важны для модели, и помогает интерпретировать чёрный ящик ансамблей, а также может использоваться для последующего feature engineering.
