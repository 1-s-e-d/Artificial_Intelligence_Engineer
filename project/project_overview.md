# Regression Modeling — LightGBM Forecasting: Обзор проекта

---

## 1. Предметная область и пользователи

Проект направлен на прогнозирование почасового электропотребления для энергосбытовых и генерирующих компаний. Пользователями сервиса выступают операторы краткосрочного планирования, аналитики и трейдеры, ответственные за формирование заявок на оптовом рынке электроэнергии и управление собственными генерирующими мощностями.

Наличие точного прогноза на день вперёд позволяет заблаговременно определить необходимый объём производства или закупок электроэнергии, избегая как дефицита, так и избыточной генерации.

---

## 2. Постановка задачи машинного обучения

Формально задача сводится к **многомерной регрессии временного ряда с экзогенными переменными**. Требуется построить модель, которая на основе исторических данных о потреблении и внешних факторов (погода, астрономические параметры, календарные признаки) выдаёт почасовой прогноз целевой переменной `Ypowerconsumption` на **24 шага вперёд**.

Таким образом, модель одновременно прогнозирует суточный профиль нагрузки, что соответствует бизнес-требованиям планирования на день вперёд. В дальнейшей перспективе рассматривается каскадная архитектура: первая модель даёт базовый прогноз, вторая — корректирует его для снижения систематической ошибки.

---

## 3. Цели и целевые метрики

**Основная бизнес-цель** — сокращение издержек, связанных с перепроизводством или непокрытием спроса.

Ключевые метрики качества модели:

| Метрика | Описание |
|---------|----------|
| **MAPE** | Mean Absolute Percentage Error — интуитивно понятная мера точности в процентах |
| **WAPE** | Weighted Absolute Percentage Error — робастная к масштабу метрика, взвешенная по суммарному потреблению |
| **R²** | Коэффициент детерминации — оценка объясняющей способности модели относительно средней нагрузки |

**Практический эффект** ожидается в виде экономии за счёт минимизации дисбалансов: снижение перепроизводства непосредственно уменьшает расходы на топливо и штрафы за отклонения.

---

## 4. Ограничения и риски

Для стабильной работы модели требуется качественный обучающий набор данных продолжительностью **не менее двух лет**, охватывающий все сезонные режимы.

Ключевые риски:

- **Неточность прогноза** может привести к перепроизводству и прямым финансовым потерям.
- **Гетероскедастичность и нестационарность** исходного ряда требуют аккуратного выбора модели и регулярной перевалидации.
- **Конфиденциальность данных**: внутренняя информация компании не подлежит разглашению, что накладывает ограничения на публикацию результатов и повторное использование датасета.
- **Ограниченная интерпретируемость** сложных ансамблевых моделей может снизить доверие операторов; поэтому предпочтение отдаётся моделям с возможностью анализа важности признаков.
- **Задержки инференса**: прогноз должен быть готов до начала планового периода (за несколько часов до суток поставки), поэтому время отклика сервиса критично.

## 6. Пайплайн загрузки, очистки и проверки качества данных

Пайплайн предобработки реализован на Python (`pandas`, `numpy`, `statsmodels`) и включает следующие этапы:

1. **Загрузка** из Excel-файла (`data/dataset.xlsx`) через `pandas.read_excel`.
2. **Проверка целостности**: отсутствие дубликатов — подтверждено; пропуски — полностью исключены на этапе предварительной фильтрации.
3. **Идентификация выбросов** методом межквартильного размаха (1.5×IQR) и их удаление: обнаружено **11 выбросов** (0.04% выборки).
4. **Приведение типов** и сортировка по временной метке.
5. **Автоматический отчёт качества** — формируется в `avto_test_py/`, включает проверки:
   - стационарности (ADF, KPSS),
   - мультиколлинеарности (VIF),
   - гетероскедастичности (Бреуш-Паган, Уайт),
   - сезонной структуры.

Такой контроль гарантирует, что в модель не попадают данные, нарушающие фундаментальные предположения регрессионного анализа. Основная реализация — `utils/model_data_preanalysis_diff.py`, запуск через `utils/run_preanalysis_diff.py`.

## 8. Реализация прогнозных моделей

В ходе экспериментов протестированы несколько классов моделей:

| Модель | Результат |
|--------|-----------|
| Линейная регрессия (Ridge/L2) | Не улавливает нелинейные взаимодействия метео и суточной сезонности |
| Решающее дерево | Склонно к переобучению |
| Random Forest | Лучше, но уступает бустингу |
| **LightGBM** | **Оптимальный баланс точности и устойчивости** |

**LightGBM** — фреймворк градиентного бустинга (Microsoft), строит ансамбль деревьев по принципу **leaf-wise**, использует гистограммный алгоритм и GOSS. Ключевые преимущества для данной задачи:

- Эффективная работа с большими датасетами.
- Учитывает нелинейные зависимости.
- Хорошо работает с гетероскедастичными данными.
- Поддерживает робастные функции потерь.
- Даёт интерпретируемые feature importance.

Для построения **доверительных интервалов прогноза** используется библиотека **MAPIE** (Model Agnostic Prediction Interval Estimator), реализующая метод conformal prediction поверх LightGBM. Это позволяет получать калиброванные интервалы неопределённости без дополнительных предположений о распределении ошибок.

Модель обучается на лаговых значениях потребления, календарных и погодных признаках для прогноза на 24 часа вперёд. В планах — дополнение LightGBM второй корректирующей моделью на остатках.

Основные скрипты: `avto_test_py/tune_allday_gpu.py`, `test/test-34.py`.

## 9. Построение Train/Val/Test и кросс-валидация

Ввиду временной природы данных применяется **временно́е разбиение** (без перемешивания):

- **Train**: исторические данные до заданной даты (`tune_start` / `train_start` из `s_config.json`).
- **Validation**: период `tune_start` — `tune_end` (используется при тюнинге гиперпараметров через Optuna).
- **Test**: период `forecast_start` — `forecast_end` (финальная оценка качества прогноза).

Стратегия скользящего окна (`train_days`, `max_windows`, `window_step_days`) реализована в `utils/SlidingWindowBifurcation.py` — модель переобучается на окне, сдвигаемом по времени, что имитирует реальный режим эксплуатации.

---

## 10. Сравнение моделей

Сравнение ведётся по метрикам MAPE, WAPE, R² на отложенном тестовом периоде. Дополнительные критерии:

- Стабильность метрик по скользящим окнам (см. `utils/SlidingWindowBifurcation.py`).
- Интерпретируемость признаков (LightGBM feature importance).
- Время инференса (бенчмарк CPU vs GPU — `avto_test_py/benchmark_device.py`).

LightGBM превосходит альтернативы по совокупности критериев, особенно по WAPE и устойчивости на нестационарных периодах.

---

## 11. Регуляризация, аугментации и подбор гиперпараметров

Применяются следующие техники:

- **Регуляризация LightGBM**: `lambda_l1`, `lambda_l2`, `min_child_samples`, `bagging_fraction`.
- **Автоматический подбор гиперпараметров** через **Optuna** (`avto_test_py/tune_allday_gpu.py`): пространство поиска задаётся в `avto_test_py/s_config.json` (поле `tune`), число испытаний — `n_trials`.
- **Поддержка GPU**: при наличии GPU-ускорения переключение осуществляется через `test/device_config.json`, который создаётся автоматически при запуске бенчмарка (`avto_test_py/benchmark_device.py`). Если файл отсутствует — используется CPU со всеми ядрами.
- **Скользящее окно обучения**: вместо статичного разбиения модель переобучается на последовательных окнах — аугментация обучающей выборки за счёт разнообразия временных периодов.

## 19. Сериализация модели

```python
import pickle

# Вариант 1: нативный формат LightGBM
model.booster_.save_model("artefakts/model.txt")

with open("artefakts/model.pkl", "wb") as f:
```

## 20. Артефакты для запуска сервиса

Для развёртывания и запуска сервиса необходимы:

1. **Docker-образ** — содержит Python-окружение, все библиотеки (LightGBM, MAPIE, pandas, numpy, scikit-learn, Flask и др.) и скрипты пайплайна.
2. **`avto_test_py/s_config.json`** — управляющий конфиг: пути, диапазоны дат, параметры модели, `n_trials`, горизонт прогноза 24 часа.
3. **`test/device_config.json`** — выбор устройства (CPU / GPU). Создаётся автоматически при запуске бенчмарка; если отсутствует — используется CPU со всеми ядрами.
4. **`data/dataset.xlsx`** — предобработанный датасет, удовлетворяющий фиксированной схеме (временна́я метка + 11 признаков + `Ypowerconsumption`).
5. **Скрипты пайплайна**:
   - Тюнинг: `avto_test_py/tune_allday_gpu.py`
   - Прогнозирование: `test/test-34.py`
   - Запуск веб-интерфейса: `run_webui.py`
6. **`artefakts/`** — директория для сохранения весов, метрик и прогнозов.

Запуск через docker-compose:
```bash
docker compose up --build
```

## 22. REST API сервиса

Сервис реализован на **Flask** (`web_interface/app.py`), доступен по адресу `http://127.0.0.1:5000`.

| Метод | URL | Описание | Тело запроса | Ответ |
|-------|-----|----------|--------------|-------|
| GET | `/api/config` | Получить конфигурацию | — | JSON-конфиг |
| POST | `/api/config` | Сохранить конфигурацию | `{"tune": {...}, "model": {...}, "forecast": {...}}` | `{"status": "ok"}` |
| GET | `/api/device_config` | Получить конфиг устройства | — | `{"device": "cpu"/"gpu"}` |
| POST | `/api/device_config` | Сохранить конфиг устройства | `{"device": "cpu"/"gpu"}` | `{"status": "ok"}` |
| GET | `/api/calendar_events` | Данные для календаря | — | JSON с датами и диапазонами |
| POST | `/api/run_tune` | Запустить тюнинг | — | `{"status": "started"}` |
| POST | `/api/run_forecast` | Запустить прогнозирование | — | `{"status": "started"}` |
| POST | `/api/run_benchmark` | Запустить бенчмарк CPU/GPU | — | `{"status": "started"}` |
| POST | `/api/stop` | Остановить активную задачу | — | `{"status": "stopped"}` |
| GET | `/api/progress` | SSE-поток логов задачи | — | `text/event-stream` |
| POST | `/api/shutdown` | Завершить сервер | — | `{"status": "shutdown"}` |

**Возможные HTTP-статусы:**
- `200 OK` — успешный запрос
- `400 Bad Request` — некорректные входные данные
- `409 Conflict` — задача уже запущена
- `500 Internal Server Error` — ошибка выполнения задачи

**Пример запроса:**
```python
import requests

BASE = "http://127.0.0.1:5000"

# Запустить тюнинг
r = requests.post(f"{BASE}/api/run_tune")
print(r.json())  # {"status": "started"}

# Получить конфигурацию
cfg = requests.get(f"{BASE}/api/config").json()
print(cfg)
```

## 23. Входные/выходные данные, ошибки и статус-коды

**Входные данные для прогноза** (`/api/run_forecast`):
- Конфиг тюнинга загружается из `s_config.json` (поле `forecast`).
- Датасет читается из `data/dataset.xlsx`.

**Выходные данные**:
- Прогноз почасового потребления на 24 часа (`Ypowerconsumption_forecast`) в формате CSV, сохраняется в `artefakts/`.
- Метрики качества (MAPE, WAPE, R²) в JSON.

**Возможные ошибки**:
- Датасет не найден → `500` + сообщение об ошибке в логах.
- Модель не обучена → `FileNotFoundError` с сообщением «Сначала запустите тюнинг».
- Задача уже запущена → `409 Conflict`.
- Некорректный диапазон дат в конфиге → `400 Bad Request`.

```python
# tests/test_smoke.py
import pytest
import requests

BASE = "http://127.0.0.1:5000"

def test_health_check():
    r = requests.get(f"{BASE}/api/health")
    assert r.status_code == 200
    assert r.json()["status"] == "ok"

def test_config_readable():
    r = requests.get(f"{BASE}/api/config")
    assert r.status_code == 200
    cfg = r.json()
    assert "tune" in cfg
    assert "forecast" in cfg

# tests/test_unit_pipeline.py
import pandas as pd
import numpy as np

def test_outlier_removal():
    data = pd.Series([100, 105, 102, 98, 500, 103])  # 500 — выброс
    q1, q3 = data.quantile(0.25), data.quantile(0.75)
    iqr = q3 - q1
    cleaned = data[(data >= q1 - 1.5 * iqr) & (data <= q3 + 1.5 * iqr)]
    assert 500 not in cleaned.values

def test_mape_calculation():
    y_true = np.array([180.0, 200.0, 160.0])
    y_pred = np.array([177.0, 198.0, 164.0])
    mape = np.mean(np.abs((y_true - y_pred) / y_true)) * 100
    assert mape < 5.0
```
